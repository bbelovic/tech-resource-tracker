Lesson 2: RxJS Submit Flows

This lesson teaches how to choose between mergeMap, switchMap, concatMap, and exhaustMap for Angular submit and request flows.

Outcome

You should be able to explain why mergeMap works in the current update branch, and when exhaustMap, concatMap, or switchMap would be a better design.

The Current Code

The update branch currently starts from the loaded resource, builds an update payload, then calls the service: 

this.editedResource.pipe(
  mergeMap((res: TechResource) => {
    const type = TechResourceType[resourceType];
    const resourceToSubmit = new TechResource(
      res.id,
      title,
      link,
      res.createdOn,
      res.status,
      type
    );
    resourceToSubmit.tags = res.tags;
    return this.techResourceService.updateResource(resourceToSubmit);
  }),
  map(() => 'Updated')
).subscribe(...);

This is a higher-order mapping case: the callback returns another observable, updateResource(...): Observable<void>. A flattening operator decides what to do with that inner observable.

The Four Choices

Operator Behavior Good for Risk
mergeMap Starts each inner observable and allows several to run at once. Writes that should all complete, where order does not matter. Double-click submit can send multiple updates concurrently.
switchMap Switches to the newest inner observable and cancels the previous one. Reads, search boxes, typeahead, route-driven reloads. Can cancel writes, which is usually wrong for save/update.
concatMap Queues inner observables and runs them one at a time in order. Every save must happen, and order matters. Can build a backlog if the user submits many times.
exhaustMap Runs the first inner observable and ignores new source values until it completes. Submit buttons where double-clicks should be ignored. Later clicks are discarded while the first request is active.

Design point

In this component, the source observable is currently editedResource, not a stream of submit clicks. That means changing mergeMap to exhaustMap here would not fully protect against double submit. A full submit-flow refactor would model submit events as a stream.

Local Improvement

For the current code shape, mergeMap is acceptable but not very communicative. Since editedResource emits once and completes, the operator choice is not doing much concurrency work. A clearer next refactor is to keep behavior the same while making the flow easier to read: 

this.editedResource.pipe(
  mergeMap((res: TechResource) =>
    this.techResourceService.updateResource(this.buildUpdatedResource(res))
  ),
  map(() => 'Updated')
).subscribe(...);

That introduces a small helper, buildUpdatedResource(res), before we redesign submit as an event stream.

Practice: Pick The Operator

A user can double-click the submit button. You want the first request to run and extra clicks to be ignored until it finishes. Which operator best matches that behavior if submit clicks are represented as a stream?
