Lesson 1: Typed Angular Services

First target: make Angular HTTP services tell TypeScript what they return. This is a small change with a large payoff: better autocomplete, fewer casts, and clearer component code.

Outcome

After this lesson, you should be able to look at a service method like postNewTechResource2(resource): Observable<Object> and rewrite it as a typed Angular method.

The Current Pattern

In this repo, TechResourceService currently has methods like this: 

postNewTechResource2(resource: TechResource): Observable<Object> {
  return this.http.post(this.url, JSON.stringify(resource), {
    headers: new HttpHeaders().set('Content-Type', 'application/json;charset=UTF-8')
  });
}

The problem is not that it fails. The problem is that Observable<Object> throws away useful information. The component later has to pretend the response is a TechResource.

The Angular 21 Direction

Angular's HTTP docs show the generic form http.get<Config>(...) and explain that omitting the generic leaves the result as Object. The same idea applies to post<T>. The type parameter is TypeScript's description of what the server returns. 

postNewTechResource(resource: TechResource): Observable<TechResource> {
  return this.http.post<TechResource>(this.url, resource, {
    headers: { 'Content-Type': 'application/json;charset=UTF-8' }
  });
}

Important

<TechResource> is a compile-time assertion. Angular does not validate the JSON at runtime. If the backend sends the wrong shape, TypeScript will not catch that after deployment.

Why This Matters

Typed service methods make the component simpler: 

this.techResourceService.postNewTechResource(resourceToSubmit)
  .pipe(map((res) => res.id > 0 ? 'Created' : 'Resource creation failed'))

TypeScript now knows res.id exists because the service returns Observable<TechResource>. You do not need map((res: TechResource) => ...) to compensate for a weak service type.
